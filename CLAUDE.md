# CLAUDE.md — HereTheyComeAgain 프로젝트 개발 지침

## 1. 성능 최적화 원칙

모든 기능 개발 시 성능을 최우선으로 고려한다.

- **오브젝트 풀링**: 반복 생성/삭제되는 오브젝트(적, 투사체, 이펙트 등)는 반드시 Object Pool을 사용할 것
- **Update 최소화**: `Update()`에서 매 프레임 실행할 필요 없는 로직은 코루틴, 이벤트, 또는 타이머로 대체할 것
- **GetComponent 캐싱**: `GetComponent()`는 `Awake()` 또는 `Start()`에서 캐싱하고, `Update()` 내에서 호출 금지
- **Draw Call 최적화**: 동일 재질의 오브젝트는 Static/Dynamic Batching 또는 GPU Instancing 활용
- **Physics 최적화**: Rigidbody 불필요 시 Kinematic 설정, 레이캐스트는 레이어 마스크 지정 필수
- **Garbage 최소화**: 런타임 중 불필요한 new 할당, LINQ, string 연산 지양 — 구조체, StringBuilder, 사전 할당 컬렉션 활용
- **프로파일링 우선**: 성능 문제 발생 시 추측이 아닌 Unity Profiler 데이터 기반으로 최적화

## 2. SOLID 원칙 준수

코드 간 과도한 의존성을 피하고 유지보수 가능한 구조를 유지한다.

- **SRP (단일 책임)**: 하나의 클래스는 하나의 역할만 담당. 예: `ZombieAI`는 AI 로직만, `ZombieHealth`는 체력 관리만
- **OCP (개방-폐쇄)**: 기존 코드 수정 없이 확장 가능하도록 추상 클래스 / 인터페이스 기반 설계
- **LSP (리스코프 치환)**: 파생 클래스는 기반 클래스를 대체할 수 있어야 함 — 예상치 못한 동작 오버라이드 금지
- **ISP (인터페이스 분리)**: 범용 인터페이스 대신 역할별 소형 인터페이스 사용 (`IDamageable`, `IPoolable`, `IInteractable` 등)
- **DIP (의존성 역전)**: 구체 클래스 직접 참조 대신 인터페이스 또는 ScriptableObject 기반 이벤트/데이터로 결합도 완화

컴포넌트 간 통신은 직접 참조보다 **UnityEvent**, **ScriptableObject 이벤트 채널**, 또는 **메시지 버스** 패턴을 우선 고려한다.

## 3. 기능 완성 후 Unity 세팅 병행

코드 작업이 완료되면, 해당 기능이 동작하는 데 필요한 Unity 에디터 설정을 함께 진행한다.

- **Inspector 설정**: 직렬화 필드(`[SerializeField]`)에 적절한 기본값 지정 및 참조 연결
- **씬 배치**: 신규 컴포넌트/프리팹이 씬에 올바르게 배치되었는지 확인
- **레이어 / 태그**: 새 오브젝트에 필요한 Layer, Tag 설정
- **Physics 설정**: Collider, Layer Collision Matrix, Rigidbody Constraints 등 물리 관련 세팅
- **프리팹 적용**: 재사용 오브젝트는 프리팹으로 등록하고 변경 사항 Apply
- **컴파일 확인**: 스크립트 수정 후 `read_console`로 컴파일 에러 없음을 확인한 뒤 에디터 세팅 진행

기능 코드 PR과 Unity 씬/프리팹/Inspector 세팅은 **같은 작업 단위**로 완결한다.

## 4. 관련 스크립트 구조 파악 (대화 첫 시작 시 1회만)

새 기능 개발 요청을 받았을 때, **그 대화에서 처음 해당 영역을 다룰 때 한 번만** 관련 스크립트의 구조와 의존 관계를 파악한다.

**파악 범위:**
- 기능과 직접 연관된 스크립트 파일만 Glob/Grep으로 탐색
- 클래스 상속 관계, 인터페이스 구현, 주요 참조(필드/메서드 호출) 확인
- 파악한 내용은 짧게 요약해서 명시 후 개발 진행
- 단, 해당 기능 관련 오류 발생 시 도저히 관련 스크립트들로 해결이 안된다면 관련 파일 외에 조사 허용
